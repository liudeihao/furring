
Repository / DAO 数据持久层

职责：
- 负责与数据源进行交互，执行最基础的增删改查操作。
- 封装所有数据访问细节。
- 它的方法通常非常原子化，比如 findById, save, delete, findByUserName。
  特点： 这一层极度专注，它的唯一目标就是“从某个地方（数据库、缓存、文件）安全高效地存取数据”。



Service 业务逻辑层

职责：
- 包含系统的核心业务逻辑和规则。例如，“用户下单后需要扣减库存”、“转账操作必须保证原子性”。
- 协调多个Repository或其它Service来完成一个完整的业务操作（即事务的边界通常在这一层）。
- 它不应该知道数据来自哪里（MySQL还是Redis？），也不应该知道请求来自HTTP API还是命令行。
  特点： 这是应用程序的心脏，是最能体现软件价值的部分。



Handler/Controller 表现层：

职责：
- 处理HTTP请求和响应（如解析JSON参数、设置状态码、返回数据格式）。
- 进行基础验证（如检查参数是否为空、格式是否正确）。
- 调用一个或多个Service来完成一个用户请求。
- 处理会话（Session）、认证（Authentication）和授权（Authorization）等Web相关逻辑。
  特点：这一层应该尽可能薄，它只应该包含与HTTP协议相关的逻辑，而不应该包含任何核心业务规则。
